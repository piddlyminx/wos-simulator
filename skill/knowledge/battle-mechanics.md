# Battle Mechanics

## Read This When

Read this before changing:

- `simulator/src/damage.ts`
- `simulator/src/damageBuckets.ts`
- `simulator/src/effectIndex.ts`
- target selection or attack ordering in `simulator/src/simulator.ts`
- troop-skill or hero-skill effect semantics

## Current Model

The TypeScript simulator is bucket based:

```text
BattleInput
  -> resolveFighter()
  -> ResolvedSkill
  -> ActiveEffect
  -> EffectIndex
  -> DamageJob
  -> bucket aggregation
  -> AttackOutcome
```

Do not use `archived/v1/**` or old Python classes as the current contract.

## Damage Jobs

A `DamageJob` is the unit of damage calculation. It contains the attacking side/unit, defending side/unit, round-start troops, and `kind`:

- `normal` - ordinary attack damage.
- `skill` - damage generated by `extra_skill_attack`.

Extra attacks are not handled by a separate legacy formula. `extra_skill_attack` effects create additional `DamageJob` objects from `trigger_damage_jobs`; each job then uses the same bucket aggregation path as normal damage, with `source.extraSkill` as the source multiplier.

## Base Damage Inputs

`calculateDamageJob()` computes:

```text
armyTerm = ceil(sqrt(attacker current unit count) * sqrt(min initial army))
```

Then bucket aggregation combines:

- static troop base stats from `troops.baseAttack`, `troops.baseLethality`, `troops.baseHealth`, `troops.baseDefense`
- player stat bonuses from `FighterInput.stats`
- passive stat effects
- dynamic active hero/troop effects
- `source.extraSkill` for skill-damage jobs

The high-level relationship is:

```text
rawDamage = numerator bucket factors / denominator bucket factors
finalKills = min(defender troops, max(0, rawDamage))
```

Trace mode exposes the exact bucket factors in `DamageEquationTrace.atomicBuckets` and `aggregationGroups`.

## Bucket Families

Use `damageBuckets.ts` as the legal effect-type list.

Important families:

- `troops.*` - raw base troop stats and current troop-count term.
- `player.*` - player/account stat bonuses from `FighterInput.stats`.
- `passive.*` - battle-start passive stat effects and input passives.
- `active.hero.*` - runtime hero modifiers.
- `active.troop.*` - runtime troop modifiers, including class-advantage troop skills.
- `type.normal.*` - applies only to normal damage jobs.
- `type.skill.*` - applies only to skill-damage jobs.
- `type.all.*` - multiplicative damage family for both normal and skill jobs.
- `source.extraSkill` - raw multiplier for `extra_skill_attack` jobs.

Static buckets are compiled into a `StaticDamageProfile` once. Dynamic buckets are looked up per job through `EffectIndex`.

## Skill Sources

`hero_skill` and `troop_skill` are data sources, not different damage equations.

Both hydrate to `ResolvedSkill`, then `ActiveEffect`, then bucket candidates or special runtime effects. Class advantage is represented by troop skills in `simulator/config/troop_skills.json`; do not add a hardcoded `class_advantage_coef`.

## Triggers And Selectors

Skill triggers are handled in `effects.ts`:

- `battle_start` fires during runtime setup.
- `turn` fires at round start and supports `every`.
- `attack` fires when an attack is declared and can filter with `trigger.source` and `trigger.target`.

Effect unit selectors resolve to `appliesTo` and `appliesVs`. Native `units.applies_vs: "all"` is invalid; use `"any"` for an unrestricted usage gate or `trigger_damage_jobs` for multi-target skill damage.

Chance lives on `trigger.probability`. Duration uses `battle`, `round`, or `attack`, with optional `delay`. Same-effect stacking is `add` by default or `max` when configured.

## Fatigue

Do not restore the old SOS/Ryo/Rapi-style fatigue term as a default Expedition mechanic. If fatigue is investigated again, add it behind an explicit experimental path and prove it with long-round controls without regressing no-hero fixtures.

## Rounding Guidance

Rounding is high risk. Before changing it, use:

- small-count controls to expose per-job or per-round cap behavior
- no-hero single-type controls
- no-hero mixed-tier controls
- before/after TypeScript testcase runner snapshots

Do not change rounding because of a hero-only mismatch.

## When A Formula Change Is Allowed

A formula or bucket aggregation change needs at least one of these:

- no-hero controls fail consistently
- mixed-tier no-hero controls identify an aggregation problem
- small-count controls identify a rounding/capping problem
- report parsing and stat extraction were ruled out
- dashboard parity improves controls without broad deterministic regressions

If controls pass and only hero cases fail, use the skill-divergence workflow instead.
